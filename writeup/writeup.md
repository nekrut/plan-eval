# Introduction

## Motivation

Frontier large language models can write code well enough to drive real scientific workflows, but the per-call cost of using them as the implementer for every routine task is hard to justify when the same models are also the ones that should be reasoning about novel problems. The question this work asks is whether the expensive part of code generation — the planning step that turns a problem statement into an unambiguous, executable recipe — can be paid for once with a frontier model, and the much cheaper part — turning that recipe into a runnable script — can then be delegated indefinitely to a small open-weight model running for free on commodity hardware already sitting in a lab. We test this on a concrete bioinformatics task (mitochondrial variant calling from short-read sequencing data) using only the kinds of machines a typical lab actually owns: a development workstation with a single consumer GPU, a Jetson edge device, an Apple-silicon laptop, and a two-GPU workstation. The software stack is pinned to standard open-source tools (BWA, samtools, LoFreq, bcftools) installed from a conda environment. If the recipe-versus-implementation split holds, the per-task marginal cost of running this kind of analysis collapses from dollars-per-call on a frontier API to electricity-and-time on hardware the lab already paid for.
